## 9. 未実装機能の詳細解説

### 9-1. ランキング・実績分析（★★ keiba-ai-proへの実装を強く推奨）

#### 何をする機能か？

AIの「過去の予測がどれだけ当たったか」を集計・可視化する機能。  
「このAIを信頼していいの？」という疑問に答えるための実績ページ。

#### SPAIAでの具体的な表示内容

- **期間別ランキング**（1ヶ月/3ヶ月/6ヶ月/12ヶ月）
  - 本命印の単勝的中率ランキング
  - 本命印の複勝的中率ランキング
  - 単勝回収率ランキング・複勝回収率ランキング
- **コース・距離・馬場別の成績内訳**（ダイヤモンド限定）
  - 例：「芝2000m・良馬場の時だけ単勝的中率60%を超える」といった傾向分析
- **万馬券的中レース一覧**（直近で万馬券を取ったレースのリスト）
- **高回収率レース一覧**

#### keiba-ai-pro での実装イメージ

現在 `prediction_log` テーブルに以下が蓄積されている：
- `race_id`, `race_date`, `predicted_rank`, `win_probability`, `odds`, `model_id`
- 実際の着順（`race_results_ultimate` からJOIN）

これを集計するだけで以下が作れる：

```sql
-- 予測1位の単勝的中率（モデル別・期間別）
SELECT
  model_id,
  COUNT(*) AS total,
  SUM(CASE WHEN actual_finish = 1 THEN 1 ELSE 0 END) AS win_count,
  ROUND(100.0 * SUM(CASE WHEN actual_finish = 1 THEN 1 ELSE 0 END) / COUNT(*), 1) AS win_rate,
  ROUND(AVG(CASE WHEN actual_finish = 1 THEN odds * 100 ELSE 0 END), 0) AS roi
FROM prediction_log
WHERE predicted_rank = 1
  AND race_date >= date('now', '-90 days')
GROUP BY model_id
ORDER BY win_rate DESC;
```

**実装ページ案：** `/prediction-history` ページに「サマリータブ」を追加

### 9-2. IPAT連携（⚠️ 実装コスト高・優先度低）

#### 何をする機能か？

JRAのインターネット馬券購入システム「即PAT（IPAT）」と連携し、  
アプリ内で選んだ買い目をIPATの投票画面に自動で入力する機能。

#### 具体的な動作（SPAIAの場合）

1. ユーザーがSPAIA内でIPATのログイン情報を登録
2. AI推奨の買い目（例：「馬連 3-7 1,000円」）を選ぶ
3. ボタン1つでIPATの注文確認画面まで遷移する
4. 最終確認はIPATサイト上でユーザーが行う

#### なぜ実装コストが高いか？

- JRAは公式APIを外部公開していない
- IPATのフォームをPOSTリクエストでシミュレートするか、  
  ブラウザ拡張機能（Chrome Extension）として実装する必要がある
- JRAの利用規約上、自動投票ツールは**グレーゾーン**（利用規約違反リスクあり）
- セッション管理・CSRF対策が複雑

#### keiba-ai-pro での代替案

IPATへの**自動投票ではなく**、以下の軽実装で十分：

```
「IPATメモ出力」ボタン
→ 推奨買い目を「馬連 3-7 / 3連複 BOX 3-5-7-9 / 計3,200円」
  という形式でクリップボードにコピーできるテキスト出力
```

### 9-3. ポイントシステム（📋 keiba-ai-pro では不要・個人利用なら無意味）

#### 何をする機能か？

サービス内仮想通貨として「ポイント」を発行し、特定コンテンツの閲覧に使用する課金モデル。

#### SPAIAでの使われ方

| ポイント取得 | ポイント消費 |
|---|---|
| プラン登録特典（最大30,000pt） | AI厳選レース β版の閲覧（数百pt/回） |
| キャンペーン | 有料コンテンツの単品購入 |
| ログインボーナス（推定） | — |

#### 仕組み

```
users テーブル
├── point_balance: 現在の残高
├── point_history テーブル（取得/消費履歴）
│   ├── amount（正: 取得, 負: 消費）
│   ├── reason（"登録特典", "コンテンツ閲覧"等）
│   └── created_at
└── ポイントAPIエンドポイント
    ├── GET /api/points/balance
    ├── POST /api/points/consume
    └── POST /api/points/grant（管理者のみ）
```

#### keiba-ai-pro への適用可否

**個人利用ツールのため不要。** 将来的にマルチユーザー化・有料サービス化するなら検討。

### 9-4. 地方競馬対応（📋 実装可能だが工数大）

#### 何をする機能か？

JRA（中央競馬）に加えて、NAR（地方競馬）のデータ収集・予測対応。

#### 対応競馬場（NAR加盟）

| 地域 | 競馬場 |
|---|---|
| 関東 | 大井、川崎、船橋、浦和 |
| 中部 | 名古屋、金沢、笠松 |
| 関西 | 園田、姫路 |
| 中国・四国 | 高知、佐賀 |
| 北海道 | 帯広（ばんえい競馬含む） |

#### 実装に必要な追加作業

1. **スクレイピング拡張**: `netkeiba.com` の地方競馬ページ（URL: `/local/`）のHTMLパーサーを追加
2. **DB拡張**: 現在の `race_results_ultimate` テーブルに `race_type = 'NAR'` カラムを追加
3. **特徴量調整**: JRAと地方競馬では以下が異なる
   - 距離体系（ダートが主体、1000m〜2400m中心）
   - 騎手が異なる（地方専属騎手 vs JRA騎手）
   - 斤量ルールの違い
4. **モデル分離 or 統合**: 地方競馬専用モデルを学習するか、JRAと統合モデルにするか選択

## 10. keiba-ai-pro のUI/UX改善提案（ストレスフリー設計）

### 現状の課題と改善案

#### 課題1: 画面遷移が遅い / データ読み込みに時間がかかる

**根本原因**: 予測APIが重い（スクレイプ＋LightGBM推論で5〜30秒）

| 改善案 | 実装方法 | 効果 |
|---|---|---|
| **スケルトンUI** | データ取得中は骨格のみ表示 | 「何も起きていない」感を解消 |
| **楽観的更新** | API応答前にUIを先行更新 | 即時フィードバック |
| **キャッシュの有効活用** | 既存の `ra-cache:` を積極表示 | 2回目以降は即時表示（実装済みだが表示強化） |
| **プリフェッチ** | レース一覧表示と同時に予測を裏でキック | レース選択→予測の待ち時間を短縮 |

#### 課題2: レース選択が面倒

**現状**: 日付 + レースIDを手動で入力 or プルダウン

| 改善案 | 実装方法 |
|---|---|
| **今日のレース一覧をトップに** | `/home` or `/race-analysis` の初期表示で今日開催の全レースを一覧表示 |
| **競馬場でフィルタ** | 東京/阪神/中山/京都 などボタンでフィルタリング |
| **「次のレース」ボタン** | レース番号順に自動遷移するNavボタン（←前 / 次→） |
| **レースステータスバッジ** | 発走前/発走中/結果確定 をリアルタイムに色分け表示 |

#### 課題3: 予測結果が読みにくい

**現状**: テーブル表示で数字が多い

| 改善案 | 実装方法 |
|---|---|
| **本命馬カード方式** | 推奨1位〜3位を大きなカードで強調表示（馬名・確率・オッズ・期待値） |
| **信号機カラー** | 期待値 >1.2=緑 / 0.8〜1.2=黄 / <0.8=赤 で直感的に判断できる |
| **買い目サマリーを先頭に** | 「馬連 3-7 BOX」などの推奨馬券を最上部に大きく表示 |
| **過去実績バッジ** | 「このモデルの直近30日 複勝的中率: 64%」をインライン表示 |

#### 課題4: 収支確認が分散している

**現状**: `/dashboard`（購入履歴）と `/prediction-history`（予測履歴）が別ページ

| 改善案 | 実装方法 |
|---|---|
| **統合ダッシュボード** | 「今日の予測」「今月の収支」「通算的中率」を1画面に集約 |
| **カレンダービュー** | 月次カレンダーで的中日（緑）・外れ日（赤）を一望 |
| **KPI数値の常時表示** | ヘッダーに「今月: +¥3,200 / 回収率118%」を小さく常時表示 |
